# Streaks & Retention — план (backlog, 2026-05-12)

Большая фича по геймификации habit-tracker'а. Цель — повысить удержание (юзеры уходят, девушка автора прямо сказала что чего-то не хватает). Аудитория — массовая по сарафанному радио, преимущественно незнакомые автору люди (не close circle, не друзья/родня); см. CLAUDE.md проекта.

Стратегия выработана через Council (Gemini 3.1 Pro + GPT-5.5), 2026-05-12. Полный council-документ остался у Айрата в `~/council/council_logs/council_2026-05-12_10-20-52.md`.

## Design intents (важно НЕ забыть)

🔴 **Утренние напоминания НЕ имеют кнопок выполнения.** Утренний reminder — это «глянь что предстоит сегодня», обзор. Отметка идёт через evening reminder, per-habit reminder или вручную через `/habits`/`/weekly`. Не плодить кнопки везде ради UX-эффекта — у каждого reminder свой смысл.

🔴 **Не копировать Duolingo leagues / leaderboards.** В близком кругу сравнивать стрики мамы (таблетки) и друга (зал) бессмысленно, конкуренция отталкивает.

🔴 **Не использовать LLM в рантайме для текстов.** Дорого, медленно, риск что бот скажет маме что-то неуместное. Массивы шаблонов в коде. LLM можно использовать **offline** — нагенерил 50 вариантов, причесал руками, положил в JSON/TS.

🔴 **Сообщения типа «мы не нужны...» при долгом пропуске НЕЛЬЗЯ.** Юзер мог быть в стрессе/болезни/завале. Лучше: «Я тут, если хочешь вернуться. Можно с одной маленькой отметки».

🔴 **Не плодить типы стриков в UI без объяснения.** Per-habit + overall + freezes + milestones — легко превратить в кашу. В UI показывать максимум 1-2 цифры одновременно.

🔴 **Grace window до 04:00 НЕ нужен.** У нас уже есть backdating до 7 дней (PR #4) — юзер может отметить задним числом через `/habits`. Это закрывает кейс «забыл отметить». Streak-расчёт читает БД с учётом backdated отметок.

## Приоритизированный план

### Шаг 1: Правильная streak-модель (фундамент)

- **Per-habit streak** считается только по due-дням привычки. Привычка "по будням" — пропуск воскресенья НЕ ломает (день не due, счётчик замирает).
- **Activity streak** ("сделал хоть что-то"): хотя бы 1 due-привычка за день. НЕ требовать 100% всех due-привычек — фрустрация будет за неделю.
- **Perfect day/week** — отдельный концепт: все due-привычки выполнены, freeze не считается perfect, missed-due ломает. Две метрики: мягкая (activity) и строгая (perfect).
- `User.timezone` уже есть. `HabitLog.date` уже хранится как локальная дата (YYYY-MM-DD).
- Не считать missed для привычки, созданной сегодня после её reminder-time.
- Paused/archived habits не участвуют в streak.

### Шаг 2: Кастомные тексты внутри существующих напоминаний

Не плодить отдельные пуши. Шаблоны разбиты по trigger'ам:
- `normal` — обычный текст напоминания
- `near_milestone` — близко к 5/10/15 дням ("ещё одна отметка — будет стрик 5 дней")
**Bucketing по дням пропуска (одобрено Эмином 2026-05-13):**
- `missed_1_day` — 1 день
- `missed_2_days` — 2 дня (автореплика missed_3_days с заменой «три»→«два»)
- `missed_3_days` — 3 дня
- `missed_few_days` — 4-6 дней, текст «несколько дней»
- `missed_week` — 7-13 дней, текст «больше недели», бот начинает беспокоиться
- `missed_long` — 14+ дней (ВСЁ дальше включая 30+), текст «очень давно не виделись», бот лично переживает

**Per-habit:**
- `habit_missed_1_day` — конкретная привычка пропущена вчера
- `habit_missed_N_days` — конкретная привычка пропущена несколько дней подряд

**Recovery удалён.** Эмин 2026-05-13: «recovery удаляем нахуй, там 30+ бесполезно уже что-то писать, в планах позже после дня 14 отключать фулл уведы челу, но пока мы не сделали там просьбу рассказать почему не пользуется ботом, не делаем».

**🔴 Backlog (после 14 дней):** в будущем — отключение всех reminder'ов после 14 дней отсутствия + одноразовая просьба рассказать через `/feedback`-механизм, почему не пользуется. Пока не реализовано, missed_long продолжает срабатывать вплоть до infinity.
- `perfect_week_ahead` — если ты сегодня закроешь все привычки, это будет perfect week

**Тон один.** Не делить тексты по profile'ам (`warm/bro/funny/strict`) — все варианты просто разные тексты на trigger, без категоризации. Решение Эмина 2026-05-12.

**Выбор шаблона**: deterministic random через seed `userId + date + trigger` — чтобы cron-ретрай не давал другой текст/дубль. Cooldown НЕ применяется (Эмин 2026-05-14: «нам кулдаун не нужен, просто рандом из всего пула»). Соответственно `MessageSent` модель отсутствует.

### Шаг 3: Milestone-поздравления + Telegram эффекты

**Стратегия текстов (вариант C из обсуждения 2026-05-12):** identity-based формулировки БЕЗ упоминания имени привычки в склонениях — у нас нет морфологического движка для русского языка, а юзеры пишут привычки как угодно («читать», «зарядка по утрам», «вода 2л»). Identity-эффект достигается через подчёркивание стабильности как факта («7 дней — это уже не случайность»), без вставки имени привычки в грамматическое предложение.

**🔴 Источник истины для конкретных milestone-текстов — `docs/streaks-texts.md`.** Все тексты согласованы с Эмином 2026-05-12, гендерно-нейтральны (без past tense глаголов с подразумеваемым «я/ты сделал/а»), используют «Название привычки» в кавычках как метку (для per-habit). Тексты ниже в этом плане — устаревшие черновики, оставлены для истории обсуждения.

**Когда отправлять:** отдельным сообщением сразу после `habit_toggle` callback который завершил день и достиг milestone'а. НЕ в reminder. Только реакция на действие юзера.

**Идемпотентность критична** (callback retry или повторное нажатие не должны слать второй раз):
```prisma
model AchievementEvent {
  id          Int      @id @default(autoincrement())
  userId      Int
  scope       String   // "overall" | "habit"
  habitId     Int?     // null для overall
  milestone   Int      // 3, 5, 7, 14, 30 ...
  achievedAt  DateTime @default(now())

  @@unique([userId, scope, habitId, milestone])
}
```

Если несколько milestone'ов совпали в один день (например 5 дней per-habit + 7 дней overall) — **один summary**, не три отдельных сообщения.

#### Telegram message effects — какие на каких milestone'ах

Эффекты подключаются параметром `message_effect_id` в `bot.api.sendMessage`. Не на каждое выполнение, только на milestone'ы. Иначе обесценится за 3 дня (классический анти-паттерн).

### Шаг 4: Streak freeze (YAZIO-like, финализировано 2026-05-13)

**Earn-логика (финал 2026-05-13):**
- Старт: **0** freezes (бесплатных заморозок нет, всё нужно копить с нуля)
- Восполнение: +1 freeze за каждые 5 дней overall-стрика (день 5, 10, 15, 20, ...)
- Max в инвентаре: **2**
- На cap'е (2/2) следующий 5-дневный milestone не даёт freeze, и НЕ доставляется уведомление (молчим)
- Уведомление о earn: отдельное сообщение, посылается при отметке которая закрыла 5-й день overall стрика (только если не на cap'е)

**Auto-apply:**
- Проверка утром: если вчера были due-привычки и НИ ОДНА не выполнена → freeze автоматически списывается из инвентаря
- Один freeze = один календарный день, покрывает overall + все per-habit стрики дня
- Юзер ничего не нажимает, всё прозрачно

**Backdating refund (Эмин 2026-05-13):**
- Если юзер задним числом отмечает день, который был покрыт freeze → день меняется на ✅, freeze **возвращается в инвентарь**
- Это означает freeze в нашей системе refundable: списан, если день остался без отметки; вернулся, если юзер всё же поставил backdated

**UI:**
- В weekly view: 🧊 вместо ⬜/🔴 на замороженном дне (при backdating потом → ✅)
- В `/stats`: «🧊 Заморозки: X/2»

**Уведомления:**
- При использовании freeze вчера → overlay-строка в утреннем reminder'е (см. `docs/reminder-texts.md` блок freeze_used). В вечернем не уведомляем.
- При получении freeze (5-й день overall) → отдельное сообщение сразу после отметки (см. блок freeze_earned). На cap'е не доставляется.
- Возврат freeze через backdating — НЕ уведомляем (Эмин 2026-05-13).
- Multiple-freeze-used кейс (несколько дней подряд закрыты freeze'ами) — отдельную обёртку не делаем, базовых freeze_used overlay'ев достаточно.

**Хранение**:
```prisma
// Расширить HabitLog
model HabitLog {
  habitId  Int
  date     String
  status   String  // "DONE" | "FROZEN" (заменяет текущее completed: Boolean)
  source   String  // "USER" | "AUTO"
  // ...
}

model FreezeUsage {
  id        Int      @id @default(autoincrement())
  userId    Int
  date      String
  reason    String   // "auto_miss" | etc
  createdAt DateTime @default(now())

  @@index([userId, date])
}
```

При миграции: `completed: true` → `status: "DONE"`, `completed: false` → удалить запись (если она вообще создавалась).

## V2 backlog

- **Recovery mode** для потерянного длинного стрика. Идея: «потерял стрик 30 дней, отметься 3 дня без пропусков — верну». НЕ возвращать как будто ничего не было — видимый repair marker, максимум 1 раз в месяц, только для значимых стриков (>= 7).

## Параллельно с фичами

Для 10-50 MAU фичами не вытащить retention в одиночку (GPT-5.5 указал). Параллельно стоит проактивно опросить неактивных юзеров через `/feedback`:
- «В какой момент перестала пользоваться?»
- «Что бесит?»
- «Что было бы полезно в напоминании?»
- «Ты не делала привычку или просто забывала отмечать?»

У нас уже есть `/feedback` infrastructure — можно сделать broadcast по `lastActiveAt > 14 days`. Это может вскрыть реальную проблему быстрее чем фичи.
